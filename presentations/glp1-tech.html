Slide 04 · PWA

PWA implementation specifics

Manifest + install

  • Standalone, portrait, neutral health/education category.
  • No medical-condition badge (platform policy compliance).
  • Custom install prompt only on engaged users.

Service worker (Workbox)

  • Pre-cache app shell + fonts.
  • Never cache /api/quiz/*, /api/eligibility, /api/match-provider.
  • Background Sync for booking confirmation form (flaky 4G).
  • Encrypted IndexedDB resume of quiz drafts (WebCrypto).
  • Strict CSP nonces; no inline scripts; HSTS preload.
Slide 05 · Funnel

Eligibility screener (≤ 90 s)

  1. Goal (lose weight, manage condition with provider, learn).
  2. Sex at birth (clinical relevance).
  3. Date of birth (gates minors).
  4. Height + weight → BMI computed server-side.
  5. Diagnoses (multi-select clinical flags).
  6. Medications + allergies.
  7. Past weight management attempts.
  8. Insurance status.
  9. State (US) — for licensed-provider matching.
  10. Phone + email + HIPAA-style consent and telehealth disclosures.

Output is a non-diagnostic suggestion + a list of matched providers + a clear next-step CTA. No outcome guarantees, ever.

PII / PHI columns column-encrypted with pgsodium. RLS default-deny. PHI reads are logged with actor + fields touched.

Slide 08 · Tracking

Server-side conversions — privacy-careful

view_learn → quiz_started → quiz_completed
→ provider_matched → slot_held → booking_confirmed
→ reminder_sent → consult_attended     // primary opt event
→ rx_started             // high-value backend
→ refill_dispensed       // LTV signal
→ dropped | optout
  • Hash SHA-256 user identifiers; never send health attributes / BMI to ad platforms.
  • Optimize Meta on booking_confirmed until ≥ 30/wk per ad set, then switch to consult_attended.
  • Google Enhanced Conversions for Leads on rx_started with value-based bidding.
  • BigQuery joins consults + postbacks + payments → 60-day cohort LTV by ad source.
Slide 09 · Reminder engine

The single biggest ROI lever

// on booking_confirmed
QStash schedule T-24h: send_sms_reminder(consult_id)
QStash schedule T-2h:  send_push_reminder(consult_id)
QStash schedule T-30m: send_sms_link(consult_id, video_link)

// on user reply "RESCH"
open /book?consult_id=...

// on user reply "STOP"
mark optout, halt all jobs
  • Quiet hours per user timezone.
  • Reminder copy A/B tested via Flagsmith feature flag.
  • Voice fallback for users without smartphones.
  • Target: lift show-up rate from ~40% baseline to ≥ 65%.

Slide 11 · Compliance

Health-vertical guardrails (white-hat)

  • BAAs signed: Twilio, Postmark/SES, Vercel (where applicable), Supabase, partner clinics.
  • pgsodium column encryption for PHI; RLS default-deny.
  • R2 with envelope-encrypted, signed-URL access (5-min default).
  • Append-only PHI access log; no PHI in any application or platform logs.
  • No "guaranteed weight loss"; no outcome promises; counsel-reviewed copy.
  • No before/after body imagery; no body-shaming hooks.
  • Visible "Educational only · Not medical advice" disclaimers everywhere.
  • DSAR + right-to-erase < 30 days.
  • Per-vertical brand domain; warmed ad accounts; pixel separation.
  • Pen test before launch + annually.

Top risks

  • FTC scrutiny on weight-loss claims → no specific outcomes, documented substantiation.
  • Show-up rate drop → reminder engine + reschedule UX + concierge fallback.
